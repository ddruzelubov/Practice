1. В названиях ссылок используются глаголы-действия вместо названий объектов  
   Плохо:  
   - `/createWorkout`  
   - `/workouts/{id}/complete`  
   - `/workouts/{id}/start`  
   - `/workouts/{id}/delete`  

2. Используются неправильные или выдуманные команды HTTP  
   - Пишут `update:` и `custom:` — таких методов не существует  
   - Удаляют тренировку через POST (`/workouts/{id}/delete`)   
   - Запускают тренировку через GET (`/workouts/{id}/start`) 

3. Придуманы несуществующие коды ответов  
   - `'100'` — возвращается как обычный ответ («Exercise creation in progress»)  
     Код 100 Continue предназначен только для промежуточного подтверждения при передаче больших тел запросов.  
     Для «принято в обработку» нужно использовать 202 Accepted.  
   - `'250'` — полностью выдуманный код, в HTTP не существует

4. Где-то используется XML вместо обычного JSON  
   Современные API почти всегда работают только с JSON. XML — это прошлый век.

5. Один и тот же функционал реализован в нескольких разных ссылках  
   Примеры:  
   - Тренировки пользователя можно получить по трём разным адресам  
   - Прогресс пользователя — по четырём разным адресам  
   Это запутывает и усложняет работу.

6. В одной спецификации встречаются все возможные стили:  
   - `/user/{userId}/workouts`  
   - `/users/{user_id}/progress/{progressId}`  
   - `/USERS/{id}/WORKOUTS`  
   - `/data/users/{userId}/data/workouts/.../data/exercises`  
   Должно быть всё в одном стиле.

7. Важные вещи передаются не в адресе, а в параметрах (?userId=…)  
   Если это идентификатор ресурса — он обязан быть в самом пути, а не в query.

8. Через обычный GET-запрос меняется состояние (например, «запустить тренировку»)  
   GET должен только читать данные, ничего не менять. Это правило безопасности.

9. Нет нормальных ответов об ошибках  
   Нет общей схемы Error, нет ответов 400, 401, 403, 404 и т.д.

10. Нет авторизации
